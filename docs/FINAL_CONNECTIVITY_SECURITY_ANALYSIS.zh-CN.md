# Mobile Codex 连通性与安全性分析（公开安全版）

更新时间：2026-04-17

这份文档保留的是可复用的方法和经验，而不是某一台私人机器的运行快照。

## 使用这份文档时的占位符约定

如果你要把自己的部署经验补充到公开仓库，请使用占位符，而不是写入真实值：

- 私有 HTTPS 入口：`<PRIVATE_HTTPS_ENTRYPOINT>`
- Tailnet IP：`<TAILNET_IP>`
- 本地应用地址：`http://127.0.0.1:3001`

不要把真实域名、真实私网地址、真实设备信息和审批痕迹写进公开文档。

## 可复用结论

### 1. 默认入口应优先是“私有 HTTPS 入口”，而不是直连私网 IP

一个稳定的私有 HTTPS 域名入口，通常比 `http://<TAILNET_IP>:3001` 这类直连方式更适合作为长期默认入口，因为它：

- 更接近稳定产品入口
- 更容易兼容浏览器和 WebView
- 更不容易把访问控制压力全部压到应用层

直连私网 IP 更适合作为排障或应急入口，而不是公开文档中的首选入口。

### 2. 运行文档必须和真实活动链路一致

如果真实运行态已经从“代理链路 A”切换到“代理链路 B”，而文档还停留在旧架构，后续排障会明显变慢。

所以公开文档里应该始终区分：

- 推荐主路径
- 备用应急路径
- 哪些组件是默认依赖
- 哪些组件只是兼容性选项

### 3. 无法读取 Tailscale LocalAPI 不等于服务下线

在 Windows 上，某些终端上下文无法读取 `tailscaled` LocalAPI。  
这类现象更适合标记为“权限盲区”，而不是直接判断为“Tailscale down”。

公开文档和状态页应该把这两类情况分开：

- 服务真的不可用
- 当前进程无权读取状态

### 4. 审批桥过期更像可用性约束，不是认证绕过

桌面审批桥如果采用“手动开启、到期自动失效”的模型，那么桥窗口过期通常意味着：

- 某些远程审批能力暂时不可用
- 但系统默认安全边界仍然成立

公开文档里应明确把这类事件描述为“可用性收口”，而不是“漏洞”。

## 推荐的公开安全排查顺序

如果要为其他开发者总结排障经验，建议按以下层次写：

1. 本地应用是否健康
2. 私有 HTTPS 入口是否健康
3. WebSocket 是否稳定
4. 当前线程是否能恢复
5. 手机端和桌面端是否能在一次刷新内收敛一致

这样写的好处是：读者先区分“入口问题”和“同步问题”，不会把所有现象都混成“手机连不上”。

## 建议保留在公开仓库里的方法论

- 优先推荐私有 HTTPS 入口
- 把直连私网 IP 降级成应急路径
- 明确记录权限盲区
- 把同步问题拆成：
  - 服务端事实源
  - 侧边栏项目快照
  - 当前线程消息视图
  - WebSocket 重连
- 让历史 incident 文档只保留方法和教训，不保留私人运行值

## 不应进入公开仓库的内容

以下内容应留在私有运行记录里，而不是这份文档里：

- 真实 tailnet 域名
- 真实 Tailscale IP
- 真实设备审批时间
- 真实 bridge 过期时间
- 真实 session、approval、device 证据
- 来自个人运行环境的原始审计 JSON 或截图

如果你要继续补这份分析，请只补“可复用经验”，不要补“私人现场记录”。
